8 卡 GPU 服务器模型推理性能测试流程

主题：通用推理框架 + AMD Radeon Pro / Radeon AI PRO + ROCm 生态专项修订版

版本：v1.0

日期：2026-05-10

适用对象：AI 平台、基础设施、模型服务、性能测试与交付团队

1. 文档目标

本文档用于指导在单机 8 卡 GPU 服务器上，对不同推理框架开展大模型推理性能测试。文档在通用 8 卡 GPU 测试方法基础上，重点补充 AMD Radeon Pro / Radeon AI PRO GPU 结合 ROCm 生态 的准入检查、框架选择、监控工具、性能测试口径和交付要求。

最终目标是形成一套可复现、可审计、可交付的测试流程，用于回答以下问题：

  1. 在同一模型、同一精度、同一输入/输出长度下，不同推理框架的吞吐、延迟和稳定性表现如何。
  2. 在满足业务 SLA 的前提下，各框架可承载的最大 RPS/QPS 或 token 吞吐是多少。
  3. 8 卡扩展效率是否合理，瓶颈来自计算、显存、PCIe/NUMA、RCCL 通信、框架调度还是模型本身。
  4. AMD Radeon Pro / Radeon AI PRO + ROCm 组合是否具备生产可用性，哪些场景适合生产，哪些场景只能作为兼容性或实验性验证。

2. 适用范围与边界

2.1 适用范围

  • 单机 8 卡 GPU 推理服务器。
  • AMD Radeon Pro / Radeon AI PRO GPU，运行 ROCm 生态。
  • 对比 vLLM、SGLang、llama.cpp/HIP、TGI、ONNX Runtime/MIGraphX 等框架。
  • 大语言模型、代码模型、多模态模型中的文本生成链路。
  • 离线吞吐测试、在线服务压测、长上下文测试、量化测试、多卡扩展测试、稳定性长跑测试。

2.2 不适用或需单独说明的范围

  • 不将 AMD Radeon Pro 结果与 NVIDIA TensorRT-LLM 结果直接作为同生态公平排名。
  • 不将 AMD Instinct MI300X/MI325X 的最佳实践直接套用到 Radeon Pro 多卡服务器。
  • 不将不同精度、不同量化、不同上下文长度、不同并行策略的结果混合排名。
  • 不用单次峰值结果替代可复现结果；所有关键结论必须至少三次复测。

3. 关键结论与推荐框架

在 AMD Radeon Pro / Radeon AI PRO + ROCm 场景下，建议将测试框架分为三类：

类别 框架 推荐定位 说明
主测框架 vLLM ROCm 生产候选 适合作为 OpenAI-compatible serving 主基线，重点测试吞吐、延迟、长上下文和多卡扩展
主测框架 SGLang ROCm 生产候选 适合高吞吐 serving、结构化输出、复杂调度、MoE 或长上下文场景
专项框架 llama.cpp HIP/ROCm 量化与轻量服务专项 适合 GGUF、INT4/INT8、本地推理和边界显存测试
兼容性框架 TGI ROCm 兼容性验证 Hugging Face 官方 ROCm 测试主要集中在 Instinct 系列，Radeon Pro 上需标明支持边界
专项框架 ONNX Runtime + MIGraphX 非 LLM 或图优化专项 适合 encoder、embedding、CV、固定图推理，不与 LLM serving 直接混排
非主测 TensorRT-LLM NVIDIA 专项 适合 NVIDIA GPU，不作为 AMD ROCm 主测框架

4. 测试指标标准

4.1 服务侧指标

指标 定义 用途
TTFT Time To First Token，首 token 延迟 衡量 prefill、排队和调度效率
ITL Inter Token Latency，token 间延迟 衡量 decode 阶段平稳性
TPOT Time Per Output Token，每输出 token 时间 与 ITL 类似，便于跨工具对齐
E2E latency 请求端到端完成耗时 用户体感延迟
Output tokens/s 输出 token 吞吐 生成服务核心吞吐指标
Total tokens/s 输入 + 输出 token 总吞吐 衡量整体计算负载
RPS/QPS 每秒请求数 容量规划指标
Goodput 满足 SLA 的有效吞吐 生产容量决策指标
Error rate 失败、超时、OOM、HTTP 5xx 比例 稳定性指标

4.2 分位数要求

所有在线服务测试必须输出：

  • P50 latency
  • P90 latency
  • P95 latency
  • P99 latency
  • 平均值、最大值、标准差

4.3 AMD ROCm 硬件指标

指标 工具 说明
GPU 利用率 amd-smi 计算资源使用情况
显存使用 amd-smi 模型权重、KV cache、runtime buffer
功耗 amd-smi 判断是否触发功耗墙
温度 amd-smi 判断是否触发热降频
GPU 频率 amd-smi 判断 clock 是否稳定
PCIe 拓扑 amd-smi topology 判断多卡通信路径
ROCm agent rocminfo 确认 GPU gfx target 与 ROCm 可见性
RCCL 通信 RCCL tests 验证 all-reduce/all-gather 性能
Kernel profile rocprof / ROCProfiler 分析 HIP kernel、memory copy、attention/GEMM 瓶颈

5. 测试矩阵设计

5.1 模型矩阵

模型级别 建议模型 测试目的
小模型 7B/8B 单卡基线、框架开销、多并发调度
中模型 14B/32B 典型生产服务负载
大模型 70B 或更大 8 卡 TP、显存与通信压力
专项模型 代码模型、Embedding、多模态模型 业务专项验证

所有模型必须记录：

  • 模型名称。
  • checkpoint commit 或 hash。
  • tokenizer 版本。
  • 权重格式。
  • 上下文长度。
  • 许可证与可用性约束。

5.2 精度与量化矩阵

精度/量化 测试原则
BF16 / FP16 主测试基线
FP8 仅与 FP8 结果横向比较，需记录 backend 与校准方式
INT8 单独分组，需记录量化方法
INT4 / AWQ / GPTQ / GGUF 单独分组，主要用于显存与吞吐收益分析

禁止将 BF16、FP8、INT8、INT4 结果混合排名。

5.3 长度矩阵

场景 输入 token 输出 token 用途
短问答 128 128 常规聊天与路由测试
标准聊天 512 128 通用在线服务基线
文档摘要 2048 256 Prefill 压力测试
长文生成 4096 512 KV cache 与 decode 压力测试
长上下文 8192+ 512 显存、attention、调度极限测试

5.4 多卡并行矩阵

配置 用途
TP=1 单卡基线
TP=2 低通信压力扩展
TP=4 中等通信压力扩展
TP=8 8 卡满配扩展
TP+PP 大模型或显存不足时使用

扩展效率计算公式：

扩展效率 = TP=N 吞吐 / (TP=1 吞吐 × N)

对于 Radeon Pro 多卡服务器，若 TP=8 吞吐低于 TP=4，必须重点排查 PCIe、NUMA、RCCL、框架调度和 batch 配置。

6. 测试前准备

6.1 硬件信息采集

执行以下命令并保存输出：

amd-smi list
amd-smi static --gpu all
amd-smi topology
rocminfo
lscpu
free -h
lsblk

应记录：

  • GPU 型号、数量、显存容量。
  • GPU gfx target。
  • PCIe 版本、lane 数、拓扑。
  • CPU 型号、NUMA 节点。
  • 内存容量与频率。
  • 磁盘类型与模型加载路径。
  • 电源、散热和机箱形态。

6.2 ROCm 兼容性准入

必须先确认以下组合是否被官方支持或可接受为实验验证：

  • GPU 型号。
  • ROCm 版本。
  • Linux 发行版与 kernel。
  • PyTorch ROCm 版本。
  • 推理框架 ROCm 支持版本。
  • 容器 runtime 与 /dev/kfd、/dev/dri 权限。

Radeon Pro 不是 Instinct 数据中心 GPU，必须显式标明支持边界。不能默认 MI300X 的测试结论适用于 Radeon Pro。

6.3 软件环境固化

必须记录：

python -V
pip freeze
docker image inspect <image>
rocminfo
amd-smi version

每个框架建议使用独立容器或独立虚拟环境。报告中必须记录：

  • 镜像名称与 digest。
  • 框架版本与 git commit。
  • ROCm、HIP、RCCL、hipBLASLt、Composable Kernel、Triton 版本。
  • Python、PyTorch、Transformers、tokenizer 版本。
  • 关键环境变量。

6.4 容器权限检查

ROCm 容器通常需要：

--device=/dev/kfd \
--device=/dev/dri \
--group-add video \
--ipc=host \
--network=host

若容器中无法看到 GPU，应优先排查 device 映射、用户组权限、ROCm runtime、IOMMU、kernel module。

7. 监控与日志方案

7.1 基础监控

建议采集：

amd-smi monitor --gpu all

如版本支持，也可导出 CSV/JSON，用于后续和压测结果对齐。

除 GPU 指标外，必须同步采集整机资源负载。推荐采集周期为 1 秒，所有监控数据必须带时间戳，便于与 benchmark 请求日志对齐。

资源 推荐工具 必采指标
CPU mpstat、pidstat、top、perf stat 可选 user/system/iowait/softirq、CPU 利用率、上下文切换、运行队列、推理服务进程 CPU
内存 free、vmstat、pidstat -r used/free/cache、swap、major/minor page fault、服务进程 RSS/VMS
磁盘 I/O iostat -xz、pidstat -d await、util、read/write MB/s、模型加载阶段 I/O 峰值
网络 sar -n DEV,TCP、ss RX/TX、TCP retransmit、连接数、accept/backlog 异常
系统调度 vmstat、pidstat -w runnable queue、blocked process、context switch、CPU 迁移
GPU amd-smi、rocprof GPU util、VRAM、功耗、温度、频率、PCIe、kernel profile

推荐后台采集命令示例：

mkdir -p logs/system
mpstat -P ALL 1 > logs/system/mpstat.log
vmstat 1 > logs/system/vmstat.log
iostat -xz 1 > logs/system/iostat.log
pidstat -urdw -p <SERVER_PID> 1 > logs/system/pidstat_server.log
amd-smi monitor --gpu all > logs/system/amd_smi_monitor.log

若使用 Prometheus/Grafana，建议同时部署 node_exporter、process_exporter 与 ROCm/AMD GPU exporter，并确保压测客户端、服务端、监控系统时间同步。

7.2 Profiling

当出现吞吐异常、TP=8 扩展效率差、TTFT 偏高或 ITL 抖动时，使用：

rocprof <command>

重点分析：

  • HIP kernel 时间分布。
  • memory copy 开销。
  • attention kernel。
  • GEMM kernel。
  • CPU 调度等待。
  • RCCL collective 通信耗时。

7.3 服务日志

每轮测试必须保存：

  • 服务启动日志。
  • benchmark 客户端日志。
  • 框架内部 warning/error。
  • OOM、timeout、HTTP 5xx。
  • ROCm/HIP/RCCL error。

8. 推理框架启动规范

8.1 vLLM ROCm

vllm serve $MODEL \
  --tensor-parallel-size 8 \
  --dtype bfloat16 \
  --max-model-len 8192 \
  --host 0.0.0.0 \
  --port 8000

需记录：

  • --tensor-parallel-size
  • --dtype
  • --max-model-len
  • GPU memory utilization
  • max batched tokens
  • max number of sequences
  • attention backend

8.2 SGLang ROCm

python3 -m sglang.launch_server \
  --model-path $MODEL \
  --tp 8 \
  --dtype bfloat16 \
  --host 0.0.0.0 \
  --port 30000

需记录：

  • TP 配置。
  • dtype。
  • attention backend。
  • scheduler 参数。
  • chunked prefill 或 cache 相关参数。

8.3 llama.cpp HIP/ROCm

./llama-server \
  -m $MODEL_GGUF \
  -ngl 999 \
  --host 0.0.0.0 \
  --port 8080

适合测试：

  • GGUF 模型。
  • INT4/INT5/INT8 量化。
  • 低显存部署。
  • 单卡或轻量多卡分层卸载。

8.4 TGI ROCm

docker run --rm -it \
  --device=/dev/kfd \
  --device=/dev/dri \
  --group-add video \
  --ipc=host \
  --network=host \
  ghcr.io/huggingface/text-generation-inference:<rocm-tag> \
  --model-id $MODEL

TGI 在 Radeon Pro 上建议作为兼容性测试项。若不稳定，应在报告中标注为不参与生产排名。

9. 标准测试流程

9.1 流程总览

8卡 AMD Radeon Pro / Radeon AI PRO + ROCm 推理性能测试流程（0到满载）
1 准入检查
确认 GPU 型号、ROCm 版本、OS/kernel、gfx target 与框架支持边界
2 环境固化
冻结驱动、ROCm、PyTorch、框架镜像、模型 checkpoint 与 tokenizer
3 ROCm 验证
rocminfo / amd-smi / torch HIP；确认 8 卡可见、状态正常
4 拓扑与通信
采集 PCIe/NUMA 拓扑，运行 RCCL 测试，识别多卡通信瓶颈
5 数据与监控
固定数据集；启动 GPU、CPU、内存、I/O、网络、服务日志同步采集
6 框架启动
vLLM、SGLang、llama.cpp/HIP、TGI 可选；健康检查与单请求验证
7 0负载基线
空载5-10分钟，记录后台噪声、CPU idle、内存常驻、GPU温度/频率
8 阶梯升载
10%→25%→50%→75%→90%→100%；每阶段观测延迟、吞吐和资源
9 满载确认
定位最大 Goodput、饱和资源、队列堆积、P99拐点和错误率边界
10 可选过载
110%-120% 验证限流、超时、错误恢复和服务退化是否可控
11 专项复测
TP=1/2/4/8、长上下文、显存边界、量化分组重复0到满载流程
12 稳定性长跑
选择Goodput点运行2-24小时，观测P99漂移、RSS增长、温度和频率
13 结论交付
输出阶段曲线、CPU/内存/GPU证据链、容量估算、推荐配置和风险清单

注：若准入失败，只输出兼容性结论；若满载阶段未满足 SLA，以最大 Goodput 点作为容量结论。

9.2 准入检查

准入检查必须在任何性能测试前完成。若准入失败，不允许输出性能排名，只能输出兼容性结论。

检查项：

  1. ROCm 可识别全部 8 张 GPU。
  2. PyTorch HIP 可识别全部 8 张 GPU。
  3. amd-smi 无异常状态。
  4. rocminfo 中 gfx target 与预期一致。
  5. 容器内外 GPU 可见性一致。
  6. RCCL 多卡通信正常。

9.3 Warmup

每个测试点必须预热：

  • 至少 3-5 分钟，或不少于 200-500 个请求。
  • 预热期间不计入正式结果。
  • 预热后检查 GPU 温度、显存和 clock 是否进入稳定区间。

9.4 单请求功能测试

并发数为 1，验证：

  • 模型可正常生成。
  • 输出 token 数符合设置。
  • TTFT、E2E latency 无明显异常。
  • 服务无 error、timeout、OOM。

9.5 Offline 极限吞吐测试

目标是测理论最大 token/s。请求尽快灌入，不模拟真实到达间隔。该场景用于评估框架和硬件的上限，不直接代表生产容量。

必须记录：

  • 总请求数。
  • 总输入 token。
  • 总输出 token。
  • 总耗时。
  • output tokens/s。
  • total tokens/s。
  • GPU 利用率与显存峰值。

9.6 Server 模式压测

使用固定 RPS 或 Poisson 到达。RPS 从低到高阶梯增长，直到：

  • P99 latency 超过 SLA。
  • error rate 超过阈值。
  • 服务 OOM。
  • GPU 发生降频、温度异常或功耗限制。
  • 队列持续堆积。

最终输出满足 SLA 的最大 Goodput。

9.7 0 到满载阶梯压测

0 到满载阶梯压测是标准流程中的核心测试。该测试用于识别不同负载阶段下，推理服务从空载、轻载、中载、高载、满载到可选过载的资源变化和性能拐点。每个框架、每个模型、每个 TP 配置都应执行该流程。

9.7.1 负载阶段定义

阶段 目标负载 建议持续时间 观测重点 阶段输出
L0 空载基线 0% 5-10 分钟 CPU idle、内存常驻、GPU 温度/频率、后台噪声 空载资源基线
L1 冒烟负载 1 个请求或并发 1 3-5 分钟 单请求 TTFT/E2E、服务进程 CPU、模型加载后显存 功能与低负载基线
L2 轻载 10%-25% 预估满载 5-10 分钟 P50/P95、CPU user/system、内存 RSS、GPU util 是否平稳 轻载效率
L3 中载 50% 预估满载 5-10 分钟 吞吐线性度、队列等待、CPU iowait/softirq、PCIe 线性扩展判断
L4 高载 75%-90% 预估满载 10 分钟 P99 抬升、上下文切换、内存 page fault、GPU 频率/功耗 SLA 安全边界
L5 满载 100% 可持续满载 10-15 分钟 最大 Goodput、饱和资源、错误率、排队深度 容量上限
L6 过载验证 110%-120%，可选 3-5 分钟 限流、超时、错误恢复、服务退化是否可控 保护与退化行为

“100% 可持续满载”不等于瞬时峰值，而是指在不违反错误率、P99 SLA、温度、功耗、显存和队列稳定性约束的情况下，可持续运行的最高负载。

9.7.2 阶梯升载方法

可选择两种升载方式之一，并在报告中说明：

  1. 固定 RPS 升载：从 0 开始，按 10% -> 25% -> 50% -> 75% -> 90% -> 100% 递增，适合在线服务容量评估。
  2. 固定并发升载：从并发 1 开始，按 1 -> 2 -> 4 -> 8 -> 16 -> 32 -> 64 -> 128 递增，适合定位框架调度和吞吐-延迟拐点。

若没有历史满载估计，先用短时探索压测获得粗略上限，再按该上限回放标准阶梯。正式报告只采用标准阶梯结果。

9.7.3 每阶段采集项

每个负载阶段必须同步记录：

  • 请求侧：RPS、并发、TTFT、ITL/TPOT、E2E latency、P50/P90/P95/P99、错误率、超时率。
  • GPU 侧：GPU util、VRAM、功耗、温度、频率、PCIe throughput、ROCm/HIP/RCCL error。
  • CPU 侧：整体 CPU、单核热点、服务进程 CPU、user/system/iowait/softirq、load average、run queue。
  • 内存侧：used/free/cache、swap、RSS/VMS、major/minor page fault、OOM killer 记录。
  • I/O 侧：磁盘 read/write、await、util；模型加载阶段和运行阶段分开记录。
  • 网络侧：RX/TX、连接数、TCP retransmit、listen backlog、客户端与服务端网络延迟。
  • 服务侧：队列长度、batch size、KV cache 使用、请求排队时间、框架 scheduler 指标。

9.7.4 升级与停止条件

满足以下条件时，才允许进入下一负载阶段：

  • 当前阶段运行满最低持续时间。
  • P99 latency 未超过 SLA。
  • error rate 低于阈值。
  • GPU 温度、频率、功耗稳定。
  • CPU 不存在持续 95% 以上单核瓶颈或异常 iowait/softirq。
  • 内存无持续增长、swap、major page fault 激增或 OOM 风险。
  • 服务队列无持续堆积。

出现以下任一情况，应停止升载并记录为瓶颈点：

  • P99 latency 超 SLA。
  • error rate 超阈值。
  • CPU run queue 持续升高且吞吐不再增长。
  • 内存 RSS 持续增长、开始 swap 或触发 OOM。
  • GPU util 接近满载但 tokens/s 不再增长。
  • GPU 温度或功耗导致频率下降。
  • PCIe/RCCL 指标显示通信成为瓶颈。
  • 服务队列持续堆积或客户端大量超时。

9.7.5 阶段性分析输出

最终报告必须给出：

  • 负载阶段 vs P99 latency。
  • 负载阶段 vs output tokens/s。
  • 负载阶段 vs CPU 利用率与 run queue。
  • 负载阶段 vs 内存 RSS、cache、swap、page fault。
  • 负载阶段 vs GPU util、VRAM、功耗、温度、频率。
  • 满足 SLA 的最大 Goodput。
  • 第一个瓶颈资源及证据链。

9.8 固定并发压测

建议并发阶梯：

1, 2, 4, 8, 16, 32, 64, 128

观察吞吐-延迟曲线，定位性能拐点。

9.9 长上下文测试

至少测试：

  • 4K 输入。
  • 8K 输入。
  • 如模型支持，测试 16K 或更高输入。

关注：

  • TTFT 是否急剧上升。
  • KV cache 是否接近显存边界。
  • 多卡 TP 是否因通信放大导致收益下降。
  • 长上下文下是否出现 OOM 或显存碎片问题。

9.10 量化专项测试

量化测试必须单独分组：

  • BF16/FP16 baseline。
  • FP8。
  • INT8。
  • INT4/AWQ/GPTQ/GGUF。

输出时必须同时给出：

  • 性能收益。
  • 显存节省。
  • 精度或质量验证方式。
  • 量化 backend。
  • 不兼容项或已知限制。

9.11 稳定性长跑

选取最佳候选配置，执行 2-24 小时稳定性测试。

观察：

  • error rate。
  • 显存是否持续增长。
  • P99 是否漂移。
  • tokens/s 是否下降。
  • GPU 温度和 clock 是否稳定。
  • 是否出现 ROCm/HIP/RCCL error。

11. 判定标准

项目 建议标准
功能正确性 所有框架通过单请求生成验证
错误率 < 0.1%；生产建议为 0
重复性 三次测试吞吐标准差 < 5%
P99 latency 必须满足业务 SLA
显存峰值 建议低于 90%-92%，需留 KV cache 和碎片余量
温度与频率 不应出现持续热降频或功耗限制
8 卡扩展 TP=8 不应低于 TP=4；若低于必须解释
日志完整性 必须保留服务日志、压测日志、监控日志
公平性 同模型、同精度、同长度、同 TP 才能横向比较

12. 结果分析方法

12.1 吞吐-延迟曲线

对每个框架绘制：

  • RPS vs P99 latency。
  • RPS vs output tokens/s。
  • 并发数 vs P99 latency。
  • 并发数 vs output tokens/s。

生产推荐点不应选择吞吐最高点，而应选择满足 SLA 且有余量的最大 Goodput 点。

12.2 多卡扩展效率

对 TP=1、2、4、8 绘制：

  • TP 数 vs tokens/s。
  • TP 数 vs 显存峰值。
  • TP 数 vs P99 latency。
  • TP 数 vs 扩展效率。

若扩展效率低，应按顺序排查：

  1. PCIe 拓扑。
  2. NUMA 绑定。
  3. RCCL collective 性能。
  4. batch size / max batched tokens。
  5. attention backend。
  6. CPU 调度与网络队列。
  7. 温度和功耗限制。

12.3 AMD ROCm 专项分析

报告中必须单独给出：

  • ROCm 版本是否为官方支持组合。
  • Radeon Pro GPU 是否为官方支持或实验支持。
  • 框架 ROCm backend 是否稳定。
  • 是否依赖特定环境变量或 workaround。
  • 是否出现 HIP、RCCL、kernel launch、memory allocation 异常。
  • 多卡性能瓶颈是否来自 PCIe 或通信。

14. 报告模板建议

正式报告建议包含以下章节：

  1. 执行摘要。
  2. 测试范围与限制。
  3. 硬件与软件环境。
  4. AMD ROCm 兼容性准入结论。
  5. 模型、框架与测试矩阵。
  6. 测试方法与指标定义。
  7. 测试结果总览。
  8. 吞吐与延迟分析。
  9. 多卡扩展效率分析。
  10. 长上下文与显存边界分析。
  11. 量化专项结果。
  12. 稳定性测试结果。
  13. 生产推荐配置。
  14. 风险与后续优化建议。
  15. 附录：命令、日志、原始数据、参考资料。

15. 风险与注意事项

  1. Radeon Pro 与 Instinct 的 ROCm 支持边界不同，必须以官方兼容矩阵为准。
  2. PCIe 多卡服务器的 8 卡 TP 扩展不一定线性，甚至可能出现 TP=8 低于 TP=4。
  3. 不同框架默认调度策略不同，必须记录所有影响 batch、KV cache、attention backend 的参数。
  4. 量化结果不能与 BF16/FP16 直接混排。
  5. 单次峰值不能作为正式结论，关键结论必须复测。
  6. 长上下文测试容易暴露显存碎片、KV cache 边界和调度策略差异，应单独分析。
  7. 若框架在 Radeon Pro 上仅为实验性支持，报告结论必须标注“实验验证，不建议直接生产使用”。
