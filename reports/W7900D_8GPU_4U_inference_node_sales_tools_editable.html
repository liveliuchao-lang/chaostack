01 / Executive Sales Positioning

销售定位总览：不是“大显存单机”，而是本地 AI 推理资源池

一句话定位：4U 8x W7900D 推理服务器适合交付给需要私有化、可控成本、多模型并发、本地数据闭环的客户，也适合纳入既有 600 台以上 IaaS 集群，在运营商高等级 IDC、成熟网络带宽和运维条件下提供推理算力资源池与裸金属算力服务器租赁。主打 32B 级模型多副本生产、70B/72B 低并发高质量服务、RAG/文档/VLM/代码助手一体化，而不是替代 Instinct/H100/H200 去跑 671B 或 1T 级高并发推理。

384GB 显存

8 张 W7900D 合计 384GB ECC GDDR6。销售表达要强调“8 个 48GB 推理单元”，不要简单说成一个天然池化的大显存。

8 路业务并发

最强打法是数据并行和任务级并行：多个 32B LLM 副本、Embedding/Reranker、VLM/OCR、代码模型、离线批处理分卡部署。

本地合规

适合金融、制造、设计院、政企、医疗、教育科研、软件研发等对数据出域敏感的客户。

成本梯度

位于“消费级拼装 GPU”与“数据中心 Instinct/H 系列”之间，适合从样板走向部门级生产。

三类业务形态：面向终端客户提供“私有化 AI 推理节点”；面向模型厂商、ISV、科研机构、企业 AI 团队提供“600+ 台规模 IaaS 推理算力资源池”；面向短期项目、模型适配、POC 和专属环境需求提供“整机裸金属 GPU 服务器租赁”。三者共用同一硬件底座，但 SLA、隔离、计费、运维责任和销售话术应分开设计。
主张 推荐说法 避免说法
算力节点定位 “一台可运行多模型、多业务队列的私有 AI 推理资源池。” “384GB 显存等同一张 384GB GPU。”
模型定位 “32B 级模型是生产主力，70B/72B 是高质量低并发，235B/671B 是验证或分流路线。” “所有开源大模型都能顺滑生产。”
ROCm 定位 “Radeon on Linux 已覆盖 PyTorch、vLLM、llama.cpp、ONNX/MIGraphX 等路线，但要逐模型 POC。” “Instinct 上的新 kernel 优化在 W7900D 上天然完全等效。”
02 / Hardware and ROCm Boundary

硬件与软件边界：先把真实能力说清楚

AMD 中国官网显示，W7900D 为 RDNA 3、48GB GDDR6、384-bit、864GB/s、ECC、FP16 矩阵 106 TFLOPS、INT8 106 TOPS、INT4 212 TOPS、295W、PCIe 4.0 x16、Triple Slot 主动散热。8 卡合计指标可以用于“资源池规模”表达，但不能等同于单模型线性加速。

指标 单卡 W7900D 8 卡合计 销售解释
显存 48GB ECC GDDR6 384GB ECC GDDR6 适合多模型和多副本；单模型跨卡需要 tensor/expert parallel 和引擎适配。
显存带宽 864GB/s 约 6.9TB/s 聚合 独立 workload 近似累加；跨卡模型并行受 PCIe/RCCL 通信影响。
FP16 矩阵 106 TFLOPS 848 TFLOPS 理论峰值 推理实际取决于 attention、GEMM kernel、batch、上下文和量化路径。
INT4 矩阵 212 TOPS 1696 TOPS 理论峰值 量化格式、算子融合、解码阶段访存瓶颈会决定实际收益。
功耗 295W TBP GPU 约 2360W，不含 CPU/内存/风扇/存储 4U 8 卡必须验证供电、气流、噪声、降频和长期稳定性。

ROCm 边界判断

软件层 适合度 8 卡节点建议
Linux + ROCm + PyTorch/Transformers 基础栈 用于模型验证、Embedding、VLM、ComfyUI、离线批处理。生产环境用容器固化版本。
vLLM ROCm 主服务栈 用于 OpenAI-compatible API、continuous batching、prefix caching、LoRA、32B 多副本服务。TP/长上下文逐模型压测。
llama.cpp HIP/ROCm 低风险备选 用于 GGUF 量化、单用户/低并发、嵌入式交付、离线知识库工具。
SGLang ROCm 前沿评估 适合复杂 agent、prefix reuse、结构化推理验证；生产前要做稳定性和 kernel 兼容性测试。
ONNX Runtime + MIGraphX 固定图稳健 适合 OCR、检测、分类、文档视觉前处理、传统 CV/NLP 模型。
W7900D 是中国市场版本，ROCm 公开兼容矩阵常以 W7900/W7900 Dual Slot 等型号列示。实际 8 卡 4U 项目必须做 OEM 整机、驱动、ROCm 可见性、mGPU、散热和长稳压测确认。

05 / 8-GPU Deployment Blueprints

8 卡部署模板：用任务级并行最大化商业价值

模板 A：企业知识库/RAG 生产节点

GPU 角色 推荐模型/服务 业务价值
GPU0-GPU3主 LLM 副本池Qwen3-32B 或 GLM-4-32B，vLLM部门多人问答、文档总结、知识库问答。
GPU4Embedding/RerankerQwen3-Embedding/BGE-M3 + Reranker提升召回和答案准确率，避免检索任务挤占主模型。
GPU5VLM/OCR 辅助Qwen2.5-VL-32B 或 OCR/ONNX 服务PDF 页面、截图、图纸、表格理解。
GPU6推理专家模型DeepSeek-R1-Distill-Qwen-32B复杂问题复核、数学、逻辑链检查。
GPU7批处理/灰度/备用离线摘要、评测、模型灰度降低线上变更风险，承接夜间批处理。

模板 B：代码助手与研发平台

GPU 分配服务适合客户注意事项
GPU0-GPU3Qwen3-Coder-30B-A3B 多副本软件公司、研发中心、芯片/EDA/工业软件团队长上下文会显著消耗显存，仓库级理解建议配合检索而不是一次塞满上下文。
GPU4代码 embedding / repo index大仓库问答、代码搜索需要增量索引、权限过滤、审计日志。
GPU5Phi-4 或 8B/14B 快速模型代码补全、测试样例、草稿响应作为低延迟模型，不承诺替代 32B 主模型质量。
GPU6-GPU7PR 摘要、单测生成、离线扫描CI/CD 流水线用队列削峰，避免影响在线问答。

模板 C：70B 高质量专家节点

当客户明确要求高质量模型，可把 8 卡拆成 2-4 个专家服务，而不是 8 个 32B 副本。示例：GPU0-GPU1 跑 70B 量化专家模型，GPU2-GPU3 跑第二个 70B/72B，GPU4 跑 embedding/rerank，GPU5 跑 32B 快速回答，GPU6 跑 VLM，GPU7 做批处理。验收重点从 QPS 转向正确率、P95、稳定性和单位任务成本。

模板 D：大模型研究/演示节点

可尝试 Qwen3-235B-A22B-Thinking-2507 的 8 卡 TP/EP 低并发评估，但必须把它定位为研究演示或高端专家服务。DeepSeek-R1 671B、Kimi K2 1T、Qwen3-Coder-480B 不建议作为 W7900D 8 卡标准交付目标。

06 / Industry Plays

行业打法：按“业务闭环”而不是“模型榜单”卖

行业 高价值场景 推荐模型栈 验收指标 销售切入语
制造/设计院/AEC 图纸问答、规范检索、BOM/工艺文档总结、招投标材料生成。 Qwen3-32B + Qwen2.5-VL-32B + BGE/Qwen3-Embedding + OCR/ONNX。 图纸/规范问答准确率、引用可追溯、批量文档处理时长。 “让图纸、规范、历史项目资料留在内网里被模型读懂。”
金融/法务/审计 合同审查、制度问答、尽调材料摘要、审计底稿辅助。 Qwen3-32B 或 GLM-4-32B + Reranker + DeepSeek-R1-Distill-32B 复核。 条款定位准确率、引用命中率、人工复核节省时间、幻觉率。 “不把敏感材料出域，也能形成可审计的 AI 助手。”
软件研发 仓库问答、代码审查、PR 摘要、单测生成、故障定位。 Qwen3-Coder-30B-A3B + Phi-4 + 代码 embedding + CI 队列。 缺陷定位命中率、单测通过率、PR 审查覆盖率、开发者采纳率。 “代码不出内网，同时让研发团队拥有自己的代码模型服务。”
教育科研 开放模型教学、多用户推理、论文/实验助手、模型对比实验。 Qwen3-32B、DeepSeek distill、Mistral、Llama、Embedding/Reranker。 并发用户数、模型切换速度、实验复现、资源隔离。 “用一台节点支撑多课程、多课题组的本地开放模型实验。”
政企/公共部门 政策问答、办事材料辅助、内部知识库、舆情材料摘要。 国产模型优先：Qwen3、GLM-4、DeepSeek distill、BGE/Qwen embedding。 内网部署、权限隔离、日志审计、引用可追溯。 “先建设可控、可审计、可迭代的本地 AI 能力。”
医疗/生命科学 病历摘要、指南检索、科研文献问答、影像报告辅助。 Qwen3-32B + 医学 RAG + Reranker + VLM/OCR。 引用准确率、敏感数据隔离、医生复核时间、合规流程。 “AI 先做辅助和检索增强，不替代专业判断。”
媒体/营销/电商 内容生成、素材理解、商品知识库、批量海报/图像生成。 Qwen3-32B/Mistral + Qwen2.5-VL + ComfyUI/视觉模型。 素材生成周期、人工修改次数、多语言输出质量。 “一台节点同时跑文案、图像、视觉理解和知识库。”
07 / Discovery Questions

售前发现问题：先问业务，再问模型，再问 SLA

业务与数据

  • 哪些数据不能出域？合同、图纸、源代码、病历、财务、政务材料分别占比多少？
  • 用户每天问多少次？高峰并发多少？是交互式问答还是批量离线任务？
  • 答案是否必须有引用来源？能否接受模型只做初稿，人工复核后使用？
  • 文档类型有哪些：PDF、扫描件、CAD 截图、Excel、网页、代码仓库、数据库？
  • 客户最想先落地的前三个业务闭环是什么？每个闭环的人工成本在哪里？

模型与质量

  • 客户是否指定模型？指定的是品牌、榜单分数、开源协议，还是已有应用适配？
  • 主力模型可接受 32B 量化吗？哪些问题必须交给 70B/235B 专家模型？
  • 是否需要中文、英文、代码、多模态、数学推理、工具调用或结构化输出？
  • 上下文长度真实需求是多少：8K、16K、32K、128K，还是“以为越长越好”？
  • 是否需要同时跑多个 LoRA、多个租户、多个模型版本？

IT 与运维

  • 机房是否支持 4U、3kW 以上整机功耗、前后风道和噪声要求？
  • 是否要求 Ubuntu/RHEL、离线安装、内网镜像源、国产化系统适配？
  • 是否已有 Kubernetes、OpenAI-compatible API 网关、权限系统和日志系统？
  • 是否需要模型审计、prompt 日志脱敏、用户权限和知识库权限联动？
  • 出现模型错误时，业务流程如何回退到人工？

采购与竞争

  • 客户当前在比较什么：消费卡、RTX 工作站卡、Instinct/H 系列、国产 AI 卡，还是云 API？
  • 采购目标是最低硬件成本、最快上线、最强性能，还是数据可控？

“能不能 Windows 部署？”“开发和演示可以看 Windows/PyTorch，但 8 卡生产推理建议 Linux + ROCm + 容器，这样 vLLM、ONNX、监控和自动化更完整。”把操作系统写入 POC 前提。
