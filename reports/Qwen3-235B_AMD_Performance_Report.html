系统性能概况
2节点16卡 AMD GPU 运行 Qwen3-235B-A22B (MoE架构)，TP=8 PP=2 跨节点张量并行。 rate 场景下系统在 r8~r16 进入饱和平台，峰值吞吐 1.615 req/s， 输出吞吐 1653.7 tok/s，总 TPM 约 186万。
TTFT 行为特征
rate_r1 TTFT P99 仅 546 ms，健康可用； rate_r2 起 TTFT P99 骤升至 58.8秒，r4 达 188秒， 说明请求队列深度远超系统实时处理能力，存在严重积压。 sync/conc 场景 TTFT 则维持在合理范围。
TPOT 稳定性
各场景 TPOT P99 极为稳定：rate 场景均在 260~318 ms， conc_c4 低至 43.3 ms，sync 场景约 158~169 ms。 说明 GPU 算力充足，瓶颈在请求调度/队列，非计算侧。
并发能力边界
conc_c4 全部完成（400/400），系统在低并发下表现优异； conc_c8 仅完成 43%（344/800），出现大量超时； conc_c16 全部失败（0/1600），系统过载崩溃。 推荐并发上限为 4，需要更高并发须优化队列策略。

rate 场景 — 持续压力测试
以固定请求速率连续发送 1000 个请求。系统在 r2 (2 req/s) 开始出现队列积压， r8 进入饱和后吞吐不再显著增长。最佳吞吐 1.615 req/s， 推荐以 r8 为生产速率上限，兼顾吞吐与服务质量。
sync 场景 — 顺序单请求
逐一发送请求，前一个完成才发下一个。最短输出 (128tok) 下吞吐最高达 8.073 req/s，输出 tok/s 最高 1259.8。 TTFT 和 TPOT 均较稳定，是验证系统单路延迟的理想场景。
conc 场景 — 并发限制测试
同时保持固定数量的并发请求。conc=4 为安全边界， 全部 400 请求完成，TTFT P99 仅 273ms，TPOT P99 仅 43.3ms。 conc=8 成功率仅 43%，conc=16 完全失败，系统并发容量有限。

结论与建议
推荐生产配置
rate_r8：请求速率 8 req/s 为推荐上限。
系统吞吐 1.587 req/s，输出 1624.8 tok/s，总 TPM 约 183万。
TPOT P99 仅 297ms，用户体验可接受。
超过此速率吞吐增益 <2%，但 TTFT 已无法恢复。
低延迟场景推荐
conc_c4：对延迟敏感的应用，推荐保持并发 ≤ 4。
TTFT P99 273ms，TPOT P99 43.3ms，
100% 完成率，无失败。适合交互式对话类应用。
TTFT 骤增原因分析
rate_r2 起 TTFT P99 从 546ms 骤升至 58.8秒（约 108倍）。
根本原因：Qwen3-235B-A22B 单次 prefill 极重（235B参数、MoE路由）， 实际系统处理能力约 ~1.3 req/s，输入速率 2 req/s 已超过容量， 导致队列深度持续增长，TTFT 随积压时间线性累加。
解决方向：引入限流/背压、chunked prefill 分片、 或调整 max_num_seqs 参数限制 KV-Cache 并发。
并发改善方向
当前 conc_c8 成功率 43%，conc_c16 完全失败。建议：
1. 调大超时时限（timeout），减少假失败
2. 开启 chunked prefill 降低单次 prefill 耗时
3. 调整 max_num_seqs 与 max_num_batched_tokens
4. 考虑请求队列优先级策略（FCFS → Shortest-First）
平台饱和区间特征
r8 至 r16 区间内，吞吐从 1.587 到 1.615 req/s，增幅仅 1.8%。 TPOT 极为稳定（P99 约 297~318ms），说明 GPU 计算侧已充分利用。 瓶颈主要在 prefill 队列调度，非 GPU 算力不足。
优化优先级
1. 高优：解决 rate>r1 的 TTFT 积压（chunked prefill / 限流）
2. 中优：提升 conc 并发容量上限（目前仅 c4 可用）
3. 低优：探索 PP=4 分流减少单节点 KV-Cache 压力
4. 监控：建立 TTFT P99 < 2s 的 SLA 告警阈值
